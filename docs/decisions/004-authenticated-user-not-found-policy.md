# Authenticated User Not Found Policy

## Context

JWT 인증 필터는 토큰에서 추출한 사용자 id로 `AppUser`를 조회하고, 사용자가 존재하는 경우에만 인증 객체를 생성한다.

따라서 인증이 필요한 API에 도달한 요청은 일반적으로 이미 유효한 사용자라고 볼 수 있다.

하지만 일부 서비스 로직에서는 인증된 사용자 id를 기반으로 새 도메인 객체를 생성해야 한다. 예를 들어 장바구니가 없는 사용자가 상품을 추가하면 `Cart`를 생성하기 위해 `AppUser` 조회가 필요하다.

이론적으로 이 시점에서 사용자가 조회되지 않는 경우는 일반적인 비즈니스 실패라기보다 데이터 불일치, 삭제된 사용자 토큰 사용, 인증 컨텍스트 불일치, 테스트 설정 오류에 가깝다.

## Decision

현재 프로젝트에서는 인증된 사용자 요청에서 서비스 계층의 사용자 조회가 실패하는 경우를 **`USER_NOT_FOUND`로 통일해 처리한다.**

다만 이를 일반적인 사용자 시나리오의 핵심 실패 케이스로 보지는 않는다.

따라서 서비스 테스트에서는 `CART_NOT_FOUND`, `CART_ITEM_NOT_FOUND`, `PRODUCT_NOT_FOUND`, `INSUFFICIENT_STOCK`처럼 서비스가 명확히 다루는 비즈니스 실패를 우선 검증하고, 인증된 사용자에 대한 `USER_NOT_FOUND`는 필요한 경우에만 방어적 케이스로 다룬다.

## Policy

- 인증 필터에서 사용자 존재 여부를 이미 확인한다.
- 인증된 사용자만 접근 가능한 API에서는 `USER_NOT_FOUND`를 일반적인 비즈니스 실패로 간주하지 않는다.
- 서비스 계층에서 사용자 조회가 필요한 경우에는 방어적으로 `USER_NOT_FOUND`를 던질 수 있다.
- 이 예외는 데이터 불일치 또는 인증 컨텍스트 불일치 상황을 표현하는 공통 예외로 사용한다.
- 학습 프로젝트 범위에서는 별도의 내부 서버 오류나 인증 컨텍스트 전용 예외를 추가하지 않는다.
- 서비스 테스트에서는 핵심 비즈니스 실패 케이스를 우선 검증하고, 인증 사용자 조회 실패 케이스는 테스트 우선순위를 낮게 둔다.

## Reason

실제 서비스라면 인증된 사용자 id로 사용자 조회가 실패하는 상황은 단순한 입력 오류가 아니라 사용자 삭제 정책, 토큰 만료 정책, 세션 무효화, 인증 서버와 DB 간 불일치 같은 운영 정책과 연결된다.

현재 프로젝트는 JWT 인증, 도메인 설계, 예외 처리, 서비스 흐름을 학습하는 목적이므로 이 상황을 세분화하기보다 기존 `USER_NOT_FOUND`로 통일하는 편이 구현 범위와 복잡도 측면에서 적절하다.

또한 서비스 코드에서 사용자 조회 실패를 방어하지 않으면 이후 로직에서 `NullPointerException` 등 의도가 불명확한 예외가 발생할 수 있다. 따라서 서비스 계층에는 방어 코드를 유지하되, 테스트에서는 정상적인 인증 사용자 흐름과 주요 도메인 실패를 중심으로 검증한다.
