# Payment Failure API

## Context

실제 커머스 서비스에서는 결제 성공과 실패가 보통 외부 결제 시스템의 callback 또는 webhook을 통해 처리된다.

하지만 현재 프로젝트는 외부 PG 연동이 없는 학습용 API 프로젝트다. 결제 성공/실패 흐름을 직접 확인하려면 이를 시뮬레이션할 API가 필요하다.

## Decision

현재 프로젝트에서는 결제 실패 처리를 일반 인증 API로 제공한다.

예상 API:

```http
PATCH /payments/{paymentId}/fail
```

이 API는 실제 운영 서비스의 사용자 기능이라기보다, 외부 결제 시스템이 없는 현재 프로젝트에서 결제 실패 상태 전이를 학습하고 테스트하기 위한 시뮬레이션 API로 본다.

## Policy

- 결제 실패는 `Payment` 상태만 `FAILED`로 변경한다.
- 주문 상태는 `PENDING_PAYMENT`로 유지한다.
- 사용자는 실패한 결제 이후 새 결제 시도를 만들 수 있다.
- 이를 위해 `Order 1 : N Payment` 관계를 사용한다.
- 같은 주문에 `PENDING` 상태의 결제가 이미 있으면 새 결제를 만들 수 없다.

## Reason

결제 실패는 주문 실패와 동일하지 않다.

결제 실패 후에도 사용자는 다른 카드나 다른 결제 수단으로 다시 결제할 수 있어야 한다. 따라서 실패한 `Payment` 기록은 남기고, `Order`는 결제 대기 상태로 유지하는 것이 더 자연스럽다.

운영 서비스라면 결제 실패 처리는 일반 사용자 API가 아니라 PG callback/webhook에서 처리하는 것이 일반적이다. 그러나 현재 프로젝트에서는 외부 PG를 구현하지 않으므로, 학습과 테스트를 위해 일반 API로 결제 실패를 시뮬레이션한다.
